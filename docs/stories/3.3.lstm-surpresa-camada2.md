## Story

**As a** usuário de inteligência de tendências,
**I want** detectar tópicos que crescem **além do previsto** por um modelo,
**so that** eu seja alertado de sinais fortes/anomalias (não só do crescimento "esperado").

## Acceptance Criteria

1. **Baseline** (persistência / média móvel / seasonal-naive) implementado e avaliado (MAE/RMSE em hold-out).
2. **LSTM** univariada treinada e avaliada **contra o baseline**, com resultados reportados honestamente (vença ou não).
3. `surprise_z = (real − previsto) / σ_resíduo` calculado; **alerta** quando `surprise_z > k` (default 2.5).
4. Limitações documentadas (séries curtas etc.). **Degrada graciosamente:** se a Camada 2 falhar, não escreve `alerts.json` e o pipeline segue só com a Camada 1.

## 📘 Guia de Desenvolvimento (para o integrante)

> Dificuldade 🔴 — **liderada pelo técnico**. Guia enxuto. Esta é a parte de Deep Learning; honestidade nas métricas vale mais que "vencer".

### 🎯 O que esta story faz
Treina um modelo (LSTM) que **prevê** quantas notícias um tópico terá amanhã. Quando a realidade **supera muito** a previsão, isso é uma "surpresa" → sinal forte/anomalia. Aqui o Deep Learning vira o **detector de surpresa** do produto.

### 🧠 Pontos de rigor (não pular)
- **Baseline é obrigatório (NFR6):** a LSTM **sempre** é comparada a um modelo simples. Reportar MAE/RMSE dos dois, honestamente — mesmo que o baseline ganhe. Isso é ciência, não fracasso.
- **Séries curtas são o limite conhecido:** documente. Não force a LSTM onde não há dado.
- **Degradação graciosa (AC4):** se a Camada 2 quebrar, o pipeline **não pode parar** — apenas não gera `alerts.json`. O dashboard já sabe lidar com isso (plano B da 3.2).
- **`scores.parquet` é evolutivo:** esta story **acrescenta** colunas (`pred, surprise_z, ...`), não recria o arquivo.

### 🪜 Passo a passo (resumido — spec §4)
1. Selecionar tópicos com série longa o suficiente; separar hold-out.
2. Baseline → prever e medir MAE/RMSE.
3. LSTM univariada (PyTorch) → treinar, prever, medir MAE/RMSE vs baseline.
4. `surprise_z = (real − pred)/(σ_resid+ε)`; `alerts.json` para `> k`. Acrescentar colunas ao `scores.parquet`. Tudo em try/except (best-effort).

### ✅ Como saber que funcionou
- Relatório com MAE/RMSE de **baseline e LSTM** lado a lado. *(AC1, AC2)*
- `scores.parquet` ganhou `pred, surprise, surprise_z, *_mae, *_rmse`. *(AC3)*
- `alerts.json` lista tópicos com `surprise_z > k` (ou não existe, se degradou). *(AC3, AC4)*
- Se você "quebrar" a Camada 2 de propósito, o pipeline termina mesmo assim. *(AC4)*

### ⚠️ Erros comuns
- **Vazamento de dados (data leakage):** não usar o futuro para prever o passado; respeitar o corte de hold-out.
- **LSTM "decorando":** séries curtas → overfitting; manter modelo pequeno e reportar honesto.

### 📚 Glossário (novos termos)
| Termo | Significado simples |
|---|---|
| LSTM | Rede neural boa para sequências/séries temporais |
| Baseline | Modelo simples de referência para comparar |
| MAE/RMSE | Medidas de erro da previsão (menor = melhor) |
| Hold-out | Parte final da série reservada só para avaliar |
| Surpresa (surprise_z) | Quanto a realidade superou a previsão |

## Dev Notes

> Contexto de `docs/architecture/adr-001-trend-score.md`, `docs/design/trend-score.md` (§4) e `docs/architecture.md` (Contratos A4, Error Handling).

- **Owner:** `src/scores/forecast.py` (estocástico/GPU — best-effort isolado). Função: `surpresa_l2(series, params) -> scores(+cols), alerts`.
- **Entrada:** `series.parquet` + `scores.parquet` (Camada 1). **Saída:** colunas extras em `scores.parquet` + `dados/scores/alerts.json`.
- **Parâmetro chave:** `k=2.5` (limiar de alerta). `σ_resid` = desvio dos resíduos na validação.
- **Degradação graciosa (Error Handling da arquitetura):** exceção na Camada 2 → log + **não** escreve `alerts.json`; pipeline segue. Único ponto onde "falhar é OK".
- **Coding Standards:** não recriar `scores.parquet` (acrescentar colunas); fixar seed onde possível.

### Testing
- **Local:** validação por **métricas** (estocástico, não unit-test de valor). Pode haver teste leve do cálculo de `surprise_z` (determinístico) dado `real, pred, σ`. Foco: relatório honesto MAE/RMSE vs baseline.
