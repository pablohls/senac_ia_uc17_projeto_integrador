## 📘 Guia de Desenvolvimento (para o integrante)

> Dificuldade 🟢. Esta story **repete** o que você já fez nas 1.2 e 1.3 para um segundo portal — ótimo para fixar o aprendizado. Glossário nas 1.1–1.3.

### 🎯 O que você vai construir
A mesma coleta de antes, mas para um **segundo portal** (Canaltech), juntando tudo na mesma tabela. Mais fontes = tendências mais confiáveis (se dois portais falam do mesmo assunto, é sinal mais forte).

### 🧠 Conceitos que você precisa entender
- **Normalização de schema:** fazer dados de fontes diferentes "caberem" no mesmo formato (as mesmas 7 colunas). Cada portal organiza do seu jeito; a gente padroniza.
- **Reuso de código:** em vez de escrever tudo de novo, generalizamos as funções das stories 1.2/1.3 para aceitar "qual portal" como parâmetro. (Princípio do projeto: REUSE > ADAPT > CREATE.)

### 🪜 Passo a passo
1. **Generalize o sitemap** (Task 1): pegue sua função da 1.2 e faça ela receber a URL do sitemap e a `fonte` como argumentos. Trate o formato do Canaltech (imprima o XML e confira a estrutura). *Por quê:* evita duplicar código.
2. **Reuse a extração** (Task 2): chame a mesma função da 1.3, passando `fonte="canaltech"`.
3. **Junte e limpe** (Task 3): use `pd.concat([corpus_atual, canaltech])` e `drop_duplicates(subset="url")`. *Por quê:* o mesmo assunto pode aparecer nos dois portais; queremos os dois registros (fontes diferentes), mas não a mesma URL duas vezes.

### ✅ Como saber que funcionou
- `corpus.parquet` agora tem linhas com `fonte = "olhar_digital"` **e** `fonte = "canaltech"`. *(AC2)*
- Nenhuma `url` repetida. *(AC3)*
- A contagem total subiu (mais artigos).

### ⚠️ Erros comuns & soluções
- **Sitemap em formato diferente:** o Canaltech pode não usar o padrão mensal — leia o XML e ajuste a leitura.
- **Schema desalinhado:** confira que as 7 colunas e seus tipos batem **exatamente** com o A1 antes de concatenar.

### 🆘 Quando me chamar

### Completion Notes List
- **Estrutura do Canaltech é diferente** (descoberta na investigação): `geral.xml` é index
  de shards `geral-N.xml` (não mensais, cobrem desde 2012); a **data vem do `<lastmod>`**
  (não da URL); a **categoria** é o 1º segmento do path. Por isso a janela é filtrada por
  `lastmod >= data_corte(meses)` em vez de selecionar sitemaps mensais.
- **REUSE > ADAPT > CREATE:** extração de texto (1.3) e schema A1 reusados **integralmente**
  via `extrair_artigos`; **adaptado** só o listing de URLs (`listar_urls_canaltech` +
  helpers determinísticos `parse_categoria_canaltech`, `data_corte`, `construir_urls_canaltech`).
  Orquestração fina em `src/coleta/canaltech.py` (`listar → extrair → mesclar`).
- **AC2/AC3:** `mesclar_corpus` faz `concat` + dedup por `url` mantendo schema A1; validado
  com amostra das duas fontes (20 artigos, 0 dup url/doc_id).
- **Resolvida recomendação do QA da 1.3:** `extrair_artigos` agora cacheia `robots.txt`
  **por host** — robusto para o lote multi-fonte (cada host tem seu robots).
- **⚠️ Qualidade de dados (documentado p/ QA, não filtrado — No Invention):**
  1. A data do Canaltech vem só do `<lastmod>`; conteúdo antigo re-editado recebe data
     recente (ex.: páginas `video` "WWDC 2017"/"E3 2016" com `data=2026`). Mais pronunciado
     que no Olhar Digital (que tem data na URL).
  2. Categorias `produto` (5.083) e `video` dominam — possível ruído (páginas de produto/
     galerias, não notícias). Filtro por categoria/qualidade fica como melhoria futura.
- **Merge completo = passo manual** (igual ao freeze da 1.3, owner): requer o `corpus.parquet`
  já congelado. Sequência da equipe:
  ```bash
  poetry run python -m src.coleta.extract     # 1.3: congela corpus (Olhar Digital)
  poetry run python -m src.coleta.canaltech   # 1.4: mescla Canaltech ao corpus
  # depois: @devops *push  (commit do corpus multi-fonte via LFS)
  ```
