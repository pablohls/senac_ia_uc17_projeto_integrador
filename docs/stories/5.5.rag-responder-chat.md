## Story

**As a** usuário do dashboard,
**I want** perguntar em linguagem natural sobre as tendências e receber uma resposta **com os artigos-fonte citados**,
**so that** eu explore o corpus conversando — e possa **verificar cada afirmação** na matéria original.

## Acceptance Criteria

1. `src/rag/responder.py`: recebe a pergunta, chama o **retriever** (Story 5.4), monta um **prompt restritivo** (*"responda SOMENTE com base nos trechos abaixo; se não houver base, diga que não encontrou"*) e chama `chat()` (Story 5.1) com `temperature=config.insight.temperature_chat`.
2. A resposta é acompanhada das **citações** (título + link dos artigos recuperados). O usuário consegue clicar e ir à fonte.
3. **Anti-alucinação (AC de rigor):** para perguntas **sem base no corpus**, o sistema **admite que não encontrou** em vez de inventar (validado com as "pegadinhas" do protocolo).
4. **Painel de chat** no `src/dashboard/app.py` via `st.chat_input`, **guardado por `tem_rag`** (endpoint LLM disponível): indisponível ⇒ o painel some, o resto do dashboard segue. **Streaming** da resposta (percepção de velocidade).
5. **Degradação graciosa:** `chat()` → `None` (LLM fora) ⇒ mensagem amigável ("assistente indisponível"), sem quebrar; retriever vazio ⇒ "não encontrei base para responder".
6. Testes: `responder` com `chat()` e `retriever` mockados — asserção de que o prompt inclui os trechos e de que resposta sem base retorna a recusa honesta. `pytest` **verde**.

## 📘 Guia de Desenvolvimento (para o integrante)

> Dificuldade 🔴 — **avançada**. É a story mais "show", e a mais perigosa: um LLM ao vivo pode **inventar**. Seu trabalho é forçá-lo a responder **só** com o que o retriever trouxe, e a **admitir** quando não sabe.

### 🎯 O que esta story faz
Fecha o RAG: a pergunta do usuário vira busca (5.4) + resposta do LLM (5.1) **citando as fontes**. Aparece um chat no dashboard. Se o LLM estiver fora, o chat simplesmente não aparece — o resto continua.

### 🧠 Pontos de rigor (não pular)
- **Prompt restritivo (AC3) — o coração da story:** deixe explícito no prompt: *"Use APENAS os trechos abaixo. Se eles não responderem, diga 'não encontrei essa informação nos artigos'."* Sem isso, o modelo alucina.
- **Sempre mostrar as citações (AC2):** a resposta sem fonte não vale. As citações são o que torna o resultado **auditável** — argumento forte na banca.
- **`tem_rag` (AC4):** mesmo padrão de `tem_camada2`/`tem_insight`. LLM fora ⇒ painel some.
- **Streaming (AC4):** um 14B leva alguns segundos; ir mostrando os tokens deixa a espera tolerável.
- **`temperature_chat` da config:** um pouco maior que 0 (fluidez), mas não alto (evita divagação).

### 🪜 Passo a passo (resumido — recommended-approach §5B)
1. `responder(pergunta)`: chama retriever → monta prompt restritivo com os trechos.
2. `chat(temperature=temperature_chat)` → resposta.
3. Anexar citações (título+link dos trechos).
4. `st.chat_input` no `app.py` sob `tem_rag`; streaming; histórico em `session_state`.
5. Tratar `None` (LLM fora) e retriever vazio.

### ✅ Como saber que funcionou
- Pergunta com base ⇒ resposta coerente **+ links** clicáveis. *(AC1, AC2)*
- Pergunta "pegadinha" (fora do corpus) ⇒ *"não encontrei nos artigos"*, sem invenção. *(AC3)*
- Ollama desligado ⇒ chat some ou avisa "indisponível"; dashboard segue. *(AC4, AC5)*
- `pytest` verde. *(AC6)*

### ⚠️ Erros comuns
- **Prompt frouxo:** sem a instrução restritiva, alucina. Teste as pegadinhas.
- **Citar artigo que não sustenta a resposta:** cite os trechos **usados**.
- **Bloquear a UI durante a geração:** use streaming; não trave o Streamlit.

### 📚 Glossário
| Termo | Significado simples |
|---|---|
| Alucinação | Quando o LLM inventa algo que não está nas fontes |
| Prompt restritivo | Instrução que limita o modelo aos trechos dados |
| Citação | Link para o artigo que sustenta a resposta |
| `tem_rag` | Flag: "o chat ao vivo está disponível?" |
| Streaming | Mostrar a resposta token a token, enquanto gera |
