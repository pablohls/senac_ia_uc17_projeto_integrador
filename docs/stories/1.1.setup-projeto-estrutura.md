## Story

**As a** equipe de desenvolvimento do TrendRadar,
**I want** a estrutura do repositório e o ambiente Python configurados com Poetry,
**so that** os 4 integrantes trabalhem sobre a mesma base modular e reprodutível, sem o problema de "só roda na máquina de um".

## Acceptance Criteria

1. Repositório contém a estrutura de pastas da arquitetura: `src/{common,coleta,pln,modelagem,scores,dashboard}/`, `config/`, `dados/{raw,processed,topics,scores}/`, `docs/`, `tests/`, `notebooks/`.
2. **Poetry** configurado (`pyproject.toml` + `poetry.lock`) com as dependências do MVP; `poetry install` conclui **sem erro**.
3. Um script de **smoke test** (`tests/smoke_test.py` ou `python -m src.common.smoke`) roda e confirma: (a) imports principais OK, (b) PyTorch enxerga a GPU (`torch.cuda.is_available()` → `True`, ou aviso explícito de modo CPU), (c) leitura/escrita Parquet OK.
4. `.gitignore` exclui artefatos pesados não versionados, `.venv`, `__pycache__`; `.gitattributes` configura **Git LFS** para `*.parquet` e `*.npy`; `git init` realizado e primeiro commit feito.
5. `config/config.yaml` existe com os parâmetros-base (fontes, `w`, `α`, `H`, `λ`, `n_min`, `k`) e é carregado/validado por `src/common/config.py` (pydantic) sem erro.
6. `README.md` documenta os 3 comandos de reprodução do demo (`poetry install` → `poetry run trendradar` → `streamlit run`).

> **Reconciliação PRD × Arquitetura:** o PRD original (AC2/AC4) citava `requirements.txt`/`environment.yml`. A arquitetura (v1.0) decidiu **Poetry** e ampliou a estrutura com `src/common` e `src/scores`. Esta story segue a **arquitetura** (fonte mais recente). Mudança registrada no Change Log.

## 📘 Guia de Desenvolvimento (para o integrante)

> Este guia é para quem vai **fazer esta story manualmente para aprender**. Vá com calma e leia o "porquê" de cada passo — é aí que está o aprendizado. Dificuldade geral: 🟢, **exceto** a parte de GPU/CUDA (🔴) — nesse ponto, chame o integrante técnico.

### 🎯 O que você vai construir
O "esqueleto" do projeto: as pastas onde cada parte vai morar, o ambiente que instala todas as ferramentas que vamos usar, e um pequeno teste que confirma "está tudo no lugar e funcionando". Nada de lógica do produto ainda — só a fundação. É como preparar a obra (terreno, fundação, ferramentas) antes de levantar as paredes.

### 🧠 Conceitos que você precisa entender
- **Repositório / Git:** uma "pasta com histórico". Toda mudança fica registrada e a equipe compartilha o mesmo conteúdo. (Busque: "o que é git e github em 5 minutos".)
- **Ambiente virtual / Poetry:** cada projeto Python precisa de um conjunto isolado de "bibliotecas" (ferramentas). O **Poetry** instala essas bibliotecas em versões exatas e travadas, garantindo que a mesma coisa rode na máquina de todos. (Busque: "poetry python para iniciantes".)
- **Dependências (`pyproject.toml` / `poetry.lock`):** a "lista de compras" das bibliotecas (`pyproject.toml`) e a "nota fiscal com as versões exatas que foram instaladas" (`poetry.lock`). O `lock` é o que evita o "na minha máquina funciona".
- **GPU / CUDA:** a placa de vídeo acelera o aprendizado de máquina. **CUDA** é o "tradutor" entre o Python e a placa NVIDIA. A versão do CUDA precisa combinar com o driver da placa — por isso essa parte é delicada.
- **Parquet:** um formato de tabela (como um Excel, mas rápido e tipado) que vamos usar para passar dados de uma fase para outra.

### 🪜 Passo a passo

**1. Crie a estrutura de pastas** (Task 1)
No terminal, dentro da pasta do projeto, crie as pastas listadas no AC1. Em cada pasta dentro de `src/`, crie um arquivo vazio chamado `__init__.py` — ele avisa o Python que aquela pasta é um "módulo" importável.
*Por quê:* separar por fase (`coleta`, `pln`, ...) é o que permite os 4 trabalharem sem pisar no código um do outro.

**2. Inicie o Poetry e adicione as dependências** (Task 2)
- Instale o Poetry (se ainda não tiver): siga o site oficial.
- Use o modelo de `pyproject.toml` que está em `docs/architecture.md` (seção "Environment com Poetry") — ele já tem a lista de bibliotecas e o truque do PyTorch.
- ⚠️ **PARE AQUI e chame o integrante técnico** para a parte do `torch`/`pytorch-cu124`: a versão `cu124` precisa bater com o driver da GPU de vocês. Se não bater, troca-se uma palavra (`cu121`, `cpu`), mas é preciso conferir junto.
- Rode `poetry install`. Pode demorar (baixa bibliotecas grandes). Se terminar sem mensagem em vermelho, deu certo.
*Por quê:* o `poetry install` cria o ambiente isolado e trava as versões no `poetry.lock`, que você vai commitar — é o coração da reprodutibilidade.

**3. Crie a configuração central** (Task 3)
- Crie `config/config.yaml` copiando os parâmetros da tabela do ADR-001 (`w=7`, `α=1`, `H=60`, `λ=1.0`, `n_min=5`, `k=2.5`) + a lista de fontes.
- Crie `src/common/config.py` com uma classe pydantic que lê esse YAML. (Há um exemplo de padrão nas Dev Notes abaixo.)
*Por quê:* deixar todos os "números mágicos" num só lugar é regra do projeto (Coding Standards) — a banca valoriza poder ajustar e justificar parâmetros.

**4. Escreva o smoke test** (Task 4)
- Crie `tests/smoke_test.py` com 3 verificações: (a) `import torch, pandas, sentence_transformers` funciona; (b) imprime se a GPU foi encontrada; (c) salva um DataFrame de teste em `.parquet` e lê de volta.
- Rode `poetry run pytest tests/smoke_test.py`.
*Por quê:* este teste é o "checklist de obra" — antes de qualquer um começar a fase dele, roda isso e sabe que o ambiente está saudável.

**5. Git, LFS e README** (Task 5)
- Crie `.gitignore` (ignora `.venv/`, `__pycache__/`, dados temporários) e `.gitattributes` (manda o Git LFS cuidar de `*.parquet` e `*.npy`, que são pesados).
- Rode `git init`, depois `git lfs install`, e faça o primeiro commit.
- Escreva o `README.md` com os 3 comandos de reprodução (estão no AC6).
*Por quê:* o LFS evita encher o repositório de arquivos gigantes; o README é o que garante que a banca (ou um integrante novo) consiga rodar o projeto do zero.

### ✅ Como saber que funcionou
- `poetry install` terminou sem erro e existe um `poetry.lock`. *(AC2)*
- `poetry run pytest tests/smoke_test.py` ficou **verde** e mostrou a linha da GPU. *(AC3)*
- `git status` mostra o projeto versionado; o primeiro commit aparece em `git log`. *(AC4)*
- Abrir `config/config.yaml` e rodar um pequeno teste de carga do `config.py` não dá erro. *(AC5)*

### ⚠️ Erros comuns & soluções
- **`poetry install` travando/lento na resolução do `torch`:** quase sempre é o `priority = "explicit"` faltando na source do PyTorch. Confira essa linha.
- **`torch.cuda.is_available()` retorna `False`:** driver/CUDA incompatível. Não trave o projeto por isso — registre como "modo CPU" e siga; é ponto de pareamento com o técnico.
- **Git reclamando do tamanho de arquivos:** faltou configurar o LFS antes de commitar os `.parquet`/`.npy`. Rode `git lfs install` e confira o `.gitattributes`.
- **`ModuleNotFoundError` ao importar `src.xxx`:** faltou o `__init__.py` na pasta, ou você não está rodando com `poetry run`.

### 🆘 Quando parar e me chamar (integrante técnico)
- Na **configuração do PyTorch/CUDA** (Passo 2) — sempre.
- Se o `poetry install` falhar com erro em vermelho que você não entende.
- Se o smoke test acusar erro que não está na lista de "erros comuns".

**Bloco da source PyTorch** (obrigatório no `pyproject.toml`):
```toml
[[tool.poetry.source]]
name = "pytorch-cu124"
url = "https://download.pytorch.org/whl/cu124"
priority = "explicit"
```
E `torch = { version = "^2.4", source = "pytorch-cu124" }`. Fallback: `cu121` / `cpu` conforme driver da GPU (should-fix levantado no architect-checklist).

**Padrão do `src/common/config.py`** (pydantic):
```python
from pydantic import BaseModel
import yaml

class TrendScoreParams(BaseModel):
    w: int = 7; alpha: float = 1.0; H: int = 60
    lambda_burst: float = 1.0; n_min: int = 5; k: float = 2.5

class Config(BaseModel):
    fontes: list[str]
    embedding_model: str
    trend_score: TrendScoreParams

def load_config(path: str = "config/config.yaml") -> Config:
    with open(path, encoding="utf-8") as f:
        return Config(**yaml.safe_load(f))
```

**Coding Standards aplicáveis** (de architecture.md): config só via objeto pydantic (sem número mágico); imports absolutos a partir de `src/`; `ruff` para lint/format.

**Script Poetry** a registrar no `pyproject.toml` (usado por stories futuras): `[tool.poetry.scripts]` → `trendradar = "run_all:main"` (o `run_all.py` será implementado em story posterior; nesta story basta um stub `def main(): ...`).
